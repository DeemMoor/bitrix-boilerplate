# delegate-codex

Ты выступаешь как **тимлид**: планируешь, делегируешь Codex-агентам, ревьюишь, мерджишь. Codex CLI (`codex exec`) делает грязную работу — твои токены тратятся только на план и ревью.

## Аргументы

Пользователь передаёт описание задачи после `/delegate-codex`. Если описания нет — спроси одним вопросом, что делать.

Опциональные параметры в описании (если пользователь укажет):
- `--agents N` — сколько codex-агентов запускать (по умолчанию 3, максимум 5)
- `--model <name>` — модель для codex (по умолчанию его дефолт)
- `--dry-run` — только спланировать, не запускать

## Шаги

### 1. Pre-flight

Параллельно проверь:
- `codex --version` — codex установлен
- `git status --porcelain` — рабочее дерево чистое (если нет — попроси пользователя закоммитить/застешить)
- `git rev-parse --abbrev-ref HEAD` — запомни текущую ветку (база для worktrees)
- `git rev-parse --show-toplevel` — корень репо

Если что-то не так — остановись и сообщи одной строкой.

### 2. Планирование

Изучи задачу и репо настолько, чтобы разбить работу на **независимые подзадачи**. Критерий независимости: подзадачи **не правят одни и те же файлы**. Если файлы пересекаются — объединяй в одну подзадачу или делай последовательно.

Сформулируй план в таком формате и **покажи пользователю до запуска**:

```
План делегирования (N агентов):
  [1] <короткое название> → файлы: a.ts, b.ts
      Промпт: <что именно сделать>
  [2] <название> → файлы: c.ts
      Промпт: ...
```

Жди подтверждения. Если `--dry-run` — на этом стоп.

### 3. Создание worktrees

Для каждой подзадачи `i`:
```
git worktree add .worktrees/codex-<i> -b codex/<slug>-<i> <base-branch>
```
`<slug>` — короткий kebab-case от названия подзадачи.

### 4. Запуск codex-агентов параллельно

Для каждого worktree запусти в фоне (`run_in_background: true`):
```
codex exec \
  --cd .worktrees/codex-<i> \
  -s workspace-write \
  --dangerously-bypass-approvals-and-sandbox \
  --skip-git-repo-check \
  "<полный промпт для подзадачи>" \
  2>&1 | tee .worktrees/codex-<i>.log
```

Если пользователь дал `--model X`, добавь `-m X`.

**Промпт для codex** должен содержать:
- Контекст: «Ты работаешь в изолированном git worktree, ветка <branch>. Делай только свою подзадачу.»
- Конкретные файлы, которые можно править
- Критерий готовности (что должно работать / какие тесты пройти)
- Просьбу в конце сделать `git add -A && git commit -m "<message>"` своих изменений

### 5. Ожидание и сбор результатов

Не поллируй — система сама уведомит о завершении фоновых задач. Когда все завершились, для каждого worktree:
- Прочитай хвост лога (`.worktrees/codex-<i>.log`, последние ~100 строк)
- Прогляди `git -C .worktrees/codex-<i> log <base-branch>..HEAD --oneline`
- Прогляди `git -C .worktrees/codex-<i> diff <base-branch>..HEAD`

### 6. Ревью (это твоя главная работа)

По каждому worktree оцени:
- **Сделано ли то, что просили?** Сверь с критерием готовности из промпта.
- **Качество кода:** мёртвый код, дублирование, нарушения стиля проекта, пропущенные edge-cases.
- **Безопасность:** не утекли ли секреты, не сломана ли валидация.
- **Тесты:** есть ли и проходят ли (запусти их если есть).

Классифицируй каждый результат:
- **OK** — мерджим как есть
- **MINOR** — поправим руками после мерджа
- **MAJOR** — отправляем codex'у на доработку с конкретными правками
- **REJECT** — выкидываем, делаем сами

### 7. Доработка / исправление

Для **MAJOR** — снова `codex exec --cd <worktree> "правки: ..."` с конкретным списком замечаний.
Для **REJECT** или **MINOR** — правь сам через Edit/Write в worktree.

### 8. Мердж

Когда все OK:
```
git checkout <base-branch>
git merge --no-ff codex/<slug>-1
git merge --no-ff codex/<slug>-2
...
```
При конфликтах — резолвь сам, не зови codex обратно.

### 9. Уборка

```
git worktree remove .worktrees/codex-<i>
git branch -d codex/<slug>-<i>
```
(Ветки оставляй, если пользователь хочет посмотреть историю — спроси.)

Финальный отчёт пользователю — 3-5 строк: что сделано, что замержено, что отправлено на доработку, что отклонено.

## Правила

- **Не делегируй то, что проще сделать самому** (правка одной строки, переименование). Минимум для делегирования — задача в 30+ строк кода или 3+ файла.
- **Не запускай больше 5 агентов** одновременно — rate-лимиты и шум в ревью.
- **Никогда не пуш** в remote без явной просьбы пользователя.
- **Если codex упал** (ненулевой exit) — покажи последние строки лога, не пытайся перезапускать молча.
- **Конфликтующие подзадачи объединяй**, не делегируй их параллельно — потом замучаешься мерджить.
