# Правила работы агента в проекте

## Общение

- Всегда отвечать пользователю на русском языке.
- Писать кратко и по делу: сначала результат, затем важные детали и проверки.
- Если правило проекта конфликтует с общими рекомендациями модели, приоритет у правила проекта.

## Границы изменений

- Не редактировать ядро Bitrix в `bitrix/`.
- Не менять пользовательские/загруженные файлы в `upload/`.
- Не менять `.env`, `local/.settings.php`, `local/php_interface/dbconn.php` и другие локальные/секретные настройки без прямого указания пользователя.
- Не удалять и не откатывать существующие изменения пользователя.
- Не делать широкие рефакторинги без явного запроса.
- Не коммитить изменения без прямой просьбы.

## Выполнение команд

- Перед изменениями проверять состояние рабочей копии через `git status`.
- Для поиска использовать `rg`, если он доступен.
- Для PHP-синтаксиса использовать `php -l <file>`.
- Для shell-скриптов использовать `bash -n <file>`.
- Если нужны проверки проекта, сначала предпочитать существующие команды из `composer.json`, `phpunit.xml.dist` и `scripts/`.
- Если команда требует Docker, сначала выяснить актуальное имя контейнера через `docker ps`; не выдумывать имя контейнера.

## Архитектура

- Следовать принципам SOLID и не смешивать ответственности между слоями.
- Контроллеры должны быть тонкими: валидация входных параметров, вызов use case, выбор HTTP-ответа.
- Бизнес-логику держать в use case/сервисах приложения.
- Доступ к данным держать в репозиториях.
- Форматирование API-ответов держать в presenter/DTO/resource-слое, если такой слой уже есть в проекте.
- Не создавать и не регистрировать зависимости внутри action-методов контроллера; регистрацию сервисов держать в composition root модуля.
- Не добавлять абстракции без практической необходимости: новая сущность должна снижать связность, убирать дублирование или фиксировать явную границу слоя.
- Конфигурацию окружения держать отдельно от кода; секреты не хардкодить.

## API и документация

- При изменении контракта API обновлять документацию/аннотации контракта, если они есть в проекте.
- Nullable-поля фактического ответа обязательно помечать как nullable в документации контракта.
- Если значение заполняется сидером, не переносить это заполнение в миграцию.
- Миграции использовать для структуры данных, сидеры — для тестового/демонстрационного контента.
- Пути к файлам и картинкам в API отдавать относительными (`/upload/...`), без схемы и хоста.

## Bitrix ORM

- Никаких legacy/процедурных API Bitrix: запрещены `CIBlockElement::GetList/GetProperty/GetByID/Add/Update/SetPropertyValuesEx`, `CIBlockSection::GetList/Add/Update`, `CIBlock::Add/Update/Delete`, `CIBlockProperty::Add`, `CUser::GetList`, `CUserTypeEntity::GetList/Add/Delete` и подобные — ни на чтение, ни на запись.
- Правило про отказ от legacy API сквозное: контроллеры, репозитории, миграции, сидеры и вспомогательные классы должны использовать объектный ORM.
- В репозиториях использовать chain API через `Table::query()`, а не `Table::getList()`.
- Для получения одной записи использовать `fetchObject()`, а не массивный `fetch()`.
- Для получения коллекции использовать `fetchCollection()`.
- Данные из ORM читать через методы объекта (`getId()`, `getName()` и т.п.), а не через массивы.
- Для чтения использовать `Table::query()->fetchObject()/fetchCollection()`.
- Для записи в инфоблочные элементы использовать `\Bitrix\Iblock\Elements\Element<ApiCode>Table::createObject()` + `setX(...)` + `save()`.
- Для записи прочих сущностей использовать соответствующие `*Table::createObject()`/`add()`/`update()`/`delete()`.
- Если кажется, что через ORM не сделать, сначала разбираться с `setSelect()`, `fetchCollection()`, связями и отсутствующими `addToSections()`, а не уходить в legacy API.
- Multiple-свойства через ORM обязательно перечислять в `setSelect()`: без `'PROP'` в селекте `$element->getProp()` может вернуть строку/null вместо `Collection`.
- Для подгрузки связанной сущности явно добавлять `'PROP.ITEM'`, `'PROP.ELEMENT'`, `'PROP.SECTION'` или другой корректный путь связи.
- Свойства типа «Справочник» (`USER_TYPE='directory'`) хранят `xml_id` в `VALUE`; Bitrix не создаёт автоматический Reference на HL-entity. Для получения данных HL-элемента делать отдельный ORM-запрос к HL-таблице по списку xml_id.
- Коды пользовательских свойств iblock не должны совпадать с системными полями элемента: `TAGS`, `NAME`, `CODE`, `ACTIVE`, `ACTIVE_FROM`, `ACTIVE_TO`, `PREVIEW_TEXT`, `PREVIEW_PICTURE`, `DETAIL_TEXT`, `DETAIL_PICTURE`, `XML_ID`, `EXTERNAL_ID`, `SORT` и т.п.
- В сомнительном случае для пользовательского свойства добавлять суффикс (`TAG_LIST` вместо `TAGS`).
- Для кеша с тегами использовать `Bitrix\Main\Data\Cache` для хранения данных и `TaggedCache` только для регистрации/очистки тегов.

## Справочники

- Простые справочники вида «xml_id + название», без иерархии, контента и привязки к публикациям, создавать как HL-блоки.
- Инфоблоки оставлять для контентных сущностей с детальным текстом, картинками, секциями и похожей структурой.

## Стиль PHP

- Соблюдать PSR-12.
- Во всех новых PHP-файлах указывать `declare(strict_types=1);`.
- Использовать строгие типы параметров, возвращаемых значений и свойств.
- Использовать `readonly` для DTO/value object/use case зависимостей, если объект не должен менять состояние после создания.
- Использовать constructor property promotion там, где это улучшает читаемость.
- Предпочитать ранние возвраты глубоким вложенным условиям.
- Импорты классов (`use`) оформлять ровной растущей лесенкой: от более коротких строк к более длинным.
- Писать код в современном стиле PHP 8.4: не добавлять лишние скобки вокруг `new`, если вызов метода или доступ к свойству можно сделать напрямую.
- Не подавлять ошибки и исключения без явной причины.
- Если исключение ловится, ответ должен соответствовать контракту API или ожидаемому поведению слоя.
- Не добавлять зависимости без необходимости.

## Тесты и проверки

- Масштаб проверок выбирать по риску изменения.
- Для точечных PHP-правок минимум проверить синтаксис изменённых PHP-файлов.
- Если меняется поведение с существующими тестами, запускать релевантные PHPUnit/Composer-проверки.
- Если проверку запустить невозможно, явно написать причину в финальном ответе.

## Запрещённые подходы

- Не писать новый код через legacy Bitrix API.
- Не смешивать SQL, форматирование ответа и бизнес-логику в контроллере.
- Не создавать сервисы через `new` внутри контроллеров/action-методов, если в проекте есть DI/composition root.
- Не хардкодить домен, абсолютные URL файлов, секреты и параметры окружения.
- Не менять ядро Bitrix и содержимое `upload/` без прямого указания пользователя.
- Не подменять архитектурное решение «быстрым костылём», если задача требует изменения слоя данных, use case или контракта API.
